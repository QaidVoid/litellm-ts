# AGENTS.md

Guidelines for AI agents working on @qaidvoid/litellm.

## Project Overview

@qaidvoid/litellm is a TypeScript SDK for [LiteLLM], the LLM proxy and gateway. It
aims to provide end-to-end type safety: literal model types, capability-narrowed
request shapes, discriminated error unions, and `Result` returns at every
public boundary instead of thrown exceptions.

The SDK is written for [Deno] first and published to JSR. A secondary npm
artifact is produced via [dnt] so Node and Bun users are not locked out.

[LiteLLM]: https://github.com/BerriAI/litellm
[Deno]: https://deno.com
[dnt]: https://github.com/denoland/dnt

## Development Workflow

### Commits

- Commit in small chunks. One logical change per commit.
- Never commit broken state. All code must format, lint, typecheck, and pass
  tests before the commit lands.
- Format and lint before commit. Run `deno task fmt` and `deno task lint`.
- Resolve every type error. Run `deno task typecheck`.

### Commit Messages

Follow conventional commits with imperative mood, one line, 72 characters or
fewer:

```
type: message
```

Types:

- `feat:` new feature
- `fix:` bug fix
- `refactor:` code restructuring without behavior change
- `test:` tests
- `docs:` documentation
- `chore:` maintenance
- `perf:` performance
- `style:` formatting only
- `ci:` CI/CD

Multi-line commit messages are discouraged. Use them only when truly
necessary, and keep the body minimal.

### Full Pre-Commit Check

```bash
deno task check
```

This runs format, lint, typecheck, and tests in sequence. It is the single
gate the project enforces before a commit lands.

### Adding Dependencies

Add dependencies to the `imports` map in `deno.json`. Prefer JSR packages over
npm. Prefer Deno standard library (`@std/*`) over third-party where reasonable.

Keep dependencies minimal. The transport, `Result` type, error types, and SSE
parser are hand-rolled on purpose. Do not pull in a library to replace any of
them without first discussing the trade-off.

## Pre-commit Checklist

Before every commit, run:

```bash
deno task check
```

Individual tasks (`fmt`, `lint`, `typecheck`, `test`) are also available for
running steps in isolation while iterating.

## Style

### General

- Default to writing no comments. Add one only when the _why_ is non-obvious.
- Document every exported symbol with JSDoc (`/** ... */`).
- Keep files small and focused. Prefer several short modules over one large
  one.
- Match the lowercase, terse, no-decoration tone of existing code.

### Type Safety

- Public API functions return `Promise<Result<T, ApiError>>` and do not throw.
- Internal helpers may throw; convert to `Result` at the public boundary using
  the `tryAsync` helper.
- Errors are a discriminated union with a `kind` tag (`network`, `http`,
  `validation`, `stream`, `auth`, etc.). Add new kinds rather than overloading
  existing ones.
- No `any`. No unchecked casts. Narrow `unknown` explicitly.
- Prefer literal types and discriminated unions over open string unions.
- Use `as const` for tuples and enum-like records.

### Modules

- One concept per file. Re-export from `mod.ts` for the public API surface.
- Internal modules go under `src/`. Generated code lives in
  `src/generated/` and is regenerated by scripts in `scripts/`, not edited by
  hand.

## Project Structure

```
.
├── mod.ts              # public re-exports
├── deno.json           # tasks, fmt, lint, compiler options
├── src/
│   ├── result.ts       # Result<T, E> type + helpers (ok, err, tryAsync, map, ...)
│   ├── error.ts        # ApiError discriminated union
│   ├── transport.ts    # typed fetch wrapper, retries, timeouts
│   ├── client.ts       # public Client surface
│   ├── models/         # model registry (capability-narrowed literal types)
│   │   └── registry.ts # generated from upstream model_prices_and_context_window.json
│   ├── api/            # public endpoint wrappers
│   │   ├── chat.ts
│   │   ├── embeddings.ts
│   │   └── ...
│   ├── streaming/
│   │   └── sse.ts      # AsyncIterable<Result<Chunk, ApiError>>
│   └── generated/      # raw types derived from upstream OpenAPI snapshot
├── scripts/            # build-time generators (model registry, openapi types)
├── tests/              # integration tests (mocked HTTP)
└── examples/           # runnable example snippets
```
